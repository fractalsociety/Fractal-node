# FractalChain — scratchpad

## Background and Motivation

FractalChain L1 testnet (PRD v0.1) is an AI-agent-first chain: HotStuff-2 consensus, hybrid Native VM + EVM (revm), unified state, FractalWork primitives as native precompiles. The PRD lives at `docs/prd.md`. Execution started with **M1: Core crate layout and determinism spike**.

**FractalWork Agent Wallet spec v2.0** lives at `docs/wallet.md` (capability tokens, budgets, tool market, receipts, staking). Phase 1 (§25.1) and §29 checklist items are being implemented in **`fractal-wallet`** in parallel with chain milestones.

## Key Challenges and Analysis

- **Determinism** is the execution layer’s highest constraint: same parent state + same ordered txs must yield identical state roots on every node.
- **M1 scope**: full Merkle Patricia Trie (EVM-compatible) is deferred to `storage`; M1 uses a **canonical commitment** over structured state (BTreeMap + borsh + keccak) so the determinism harness is meaningful before MPT lands.
- **Wallet spec**: Revocation proofs use a **sorted-leaf Merkle tree** (BLAKE3) in Phase 1 code; full sparse Merkle trie per §4.6 can replace `merkle` internals later without changing `RevocationSet` API surface much.

## Lessons

- Read `Cargo.toml` before editing workspace members; hyphenated crate dirs map to underscored package names in Rust 2021.
- Agent sandbox may lack `cargo`; user-run `cargo test` is the source of truth for compile/test until CI exists.
- **Borsh 1.5**: enums with `#[repr(u8)]` explicit discriminants require `#[borsh(use_discriminant = true)]` (or `false`) on the enum.
- `BTreeSet<(ToolClass, TeeType)>` requires `TeeType: Ord` (derive `PartialOrd, Ord` on field-less enums).
- **jsonrpsee 0.24:** there is no `http-abi` feature; use `features = ["server", "macros"]` in workspace `Cargo.toml`.
- **Producer vs RPC:** `producer_loop` must hold `Arc<Mutex<NodeInner>>` so it can read `mempool` / `state`; RPC uses `CoerceUnsized` to `SharedChain` from the same `Arc`. Do not type the producer as `SharedChain` (`dyn` loses fields).
- **libp2p request-response:** overlapping `GetTip`/`GetBlocks` requests can deliver responses out of order and break followers (e.g. duplicate height-1 apply). Serialize with an `outstanding` flag or single-pending RPC.
- **`NativeCall::try_from_slice` outside `fractal-core`:** import `borsh::BorshDeserialize` (methods are on the trait).
- **jsonrpsee / MetaMask:** `eth_call` revert uses RPC error code `3` and `data` as a JSON string value holding `0x` + hex return data (common wallet pattern).
- **Revm + `State::bump_nonce`:** For top-level `EvmCall` into contract bytecode, revm’s `transact` commits `caller_nonce + 1`. Core still calls `bump_nonce` after `execute_call`; reset the caller’s nonce to the pre-`transact` value in `RevmEngine::execute_call` after `db.commit` so the net increment is exactly one. Precompile fast path does not use revm and still relies on `bump_nonce` only.
- **jsonrpsee 0.24 + `tower-http` CORS:** `ServerBuilder::set_http_middleware` expects **`tower` 0.4** `ServiceBuilder` (jsonrpsee’s dependency). Using workspace `tower` 0.5 causes a type mismatch.
- **`borsh::to_vec` (1.5):** free function with `T: BorshSerialize` bound on the value; the trait does **not** need to be in scope at the call site. Only `BorshDeserialize` must be imported (it provides `try_from_slice`).
- **`PolicyTemplate` has no `allowed_classes` field:** §15.2's "allowed: X, Y" list is realized at capability-mint time as `Scope.tool_class_mask`, not as a caveat. Use `policy_builtins::suggested_tool_class_mask(template_id)` to materialize the mask without a schema change.
- **`fractal-cli` package layout:** `[lib] path = "src/lib.rs"` + `[[bin]] name = "fractal-wallet-cli" path = "src/main.rs"` so integration tests can call `fractal_cli::run_argv_value` directly without spawning a process (avoids cargo's binary-discovery quirks under `--test`).
- **`CapabilityToken::verify_attenuation_from_parent`:** body-only check (scope subset + time within window + `caveats_attenuate_parent`). Does not validate that the supplied issuer secret matches `parent.issuer` — callers (e.g. `cap attenuate`) must check that explicitly before signing, otherwise a child looks valid offline but its sig fails the next time `verify()` is called against `parent.issuer`.
- **Attenuating `MaxTotalSpend`:** parent may have multiple `MaxTotalSpend` caveats (rare but legal); strict child cap must be ≤ the **minimum** of all parent caps. If the parent has none, the child may add a `MaxTotalSpend` (adding restrictions is always allowed).
- **`docs/explorer.md`:** documents Ethereum `keccak(raw)` vs internal `keccak(borsh)` tx hashes and leader vs follower JSON-RPC — link from `tools/explorer` and `docs/devnet.md`.
- **`fractal_sdk::provider`:** depends on `fractal-wallet`; re-exports market + `provider_id_from_public_key` for provider operators (`crates/sdk-rust/tests/provider_sdk.rs`).
- **MVP bridge smoke:** `./scripts/wait-for-jsonrpc.sh` requires `curl`. **GitHub:** pushing files under **`.github/workflows/`** needs a PAT with **`workflow`** scope (or use the web UI); workflow YAML is vendored as **`docs/ci/mvp-bridge-smoke.workflow.yml`** for copy-install.
- **HotStuff-2 `parent_qc_hash`:** First block after genesis uses `hash_qc(&genesis_parent_qc())`; each commit advances via `next_parent_qc_hash_after_commit`; followers reject `SyncApplyError::ParentQcHash` on mismatch.
- **`BlockHeader.proposer`:** must equal `validators.expected_proposer(view)` (`view % n`); see `crates/consensus/src/validators.rs` and `NodeInner::apply_synced_block`.
- **M7-c `FRACTAL_VALIDATOR_INDEX`:** out-of-range values are **clamped to 0 with a stderr warning**, not rejected — so a stale env var on a singleton devnet never silently disables block production. Operators see `fractal-node: FRACTAL_VALIDATOR_INDEX=X ≥ validator_set_size=N; clamping to 0`.
- **M7-c mempool invariant:** the leader gate in `try_produce_one_tick` runs *before* `mempool.drain_ready_gas_budget`. Reordering this (e.g. draining to inspect priority before checking turn) would silently drop unsigned txs on non-leader ticks. The non-leader's mempool must stay intact for the next leader once vote gossip / view-change ships.
- **`apply_synced_block` advances view:** followers update `self.view = block.header.view + 1`, so multi-binary BFT-7 deployments don't need vote gossip *yet* to rotate leaders — each binary's view advances when it receives the previous leader's block via QUIC sync. Single-binary BFT-7 still stalls after view 0 (no way to advance view locally), which is intentional under M7-c.
- **`blst` 0.3 min_pk module:** Ethereum/Lighthouse convention is `blst::min_pk::*` — public keys live in G1 (48 bytes compressed) and signatures in G2 (96 bytes compressed). The PoP-variant DST `BLS_SIG_BLS12381G2_XMD:SHA-256_SSWU_RO_POP_` is required for `fast_aggregate_verify`, which trusts the caller to have rejected rogue-key attacks via proof-of-possession at validator registration time. Compiling `blst` adds 1-2 minutes to a clean build (it builds the C library via `cc`); cached after first run.
- **`AggregateSignature` byte layout preserved:** intentionally kept `[u8; 96]` so the borsh wire format of `QuorumCertificate` is identical pre- and post-M7-d-1 — only signature *semantics* change (real verify instead of placeholder). This means M7-d-1 alone doesn't break sync between old/new nodes; M7-d-6 is where the breaking change lands.
- **PBFT quorum formula:** the canonical bound is `f = floor((n - 1) / 3)`, `quorum = 2f + 1`. The `n / 3` shorthand is wrong at exact multiples of 3 (e.g. n=3 should be 1, not 3, because f=0 there). Code in `ValidatorSet::quorum_threshold` matches the canonical form.
- **`FRACTAL_VALIDATOR_SECRET_HEX` fail-soft:** malformed env values (bad hex, wrong length, off-curve scalar, **or pubkey-mismatched** vs `validators.bls_pubkey(validator_index)`) log a stderr warning instead of aborting. Pubkey-mismatched still returns the user-supplied key so the operator can observe peers rejecting their votes and fix the config; everything else falls back to the dev key. Rationale: a typo in a 64-char hex string must never take a leader offline silently.
- **`ValidatorSet::dev_bls_secret`:** intentionally only returns `Some` for the built-in `phase1_singleton` / `phase2_bft7_fixture` fixtures so future operator-provisioned sets (which won't have a recoverable secret per pubkey) return `None` and force `FRACTAL_VALIDATOR_SECRET_HEX` to be set.
- **`Vote` does NOT sign `validator_index` (revised in M7-d-4):** BLS already binds a signature to a specific signer via the public key — verifying Alice's sig against Bob's pubkey fails regardless of envelope labeling. Removing `validator_index` from the *signed body* makes every signer's canonical bytes identical, which is the prerequisite for `fast_aggregate_verify` (one msg, many pubkeys). The replay defense lives in the verify path (`Vote::verify_against_validator_set` does `bls_pubkey(validator_index).verify(sig, body)`). The M7-d-3 design that signed `validator_index` was a redundant belt-and-suspenders that would have forced the more expensive `aggregate_verify` (distinct-msg) path for QCs.
- **`VoteSignBody::sign_bytes` unwraps borsh:** the body is all fixed-size scalars + a `[u8; 32]`, so borsh serialization is infallible — `expect` is safe and avoids a `Result` on every sign call.
- **`VotePool` keyed by `(view, header_hash)` not `header_hash` alone:** two validators can vote for the same header_hash at *different views* in HotStuff-2 (during view-change retries). Keying by header_hash alone would silently merge those into a single slot and form a confused QC. The inner `BTreeMap<validator_index, Vote>` makes "validator already voted at this slot" a `contains_key` check (O(log n)) and double-vote prevention is implicit. `try_form_qc` collects **all** valid signers (not just the threshold), so a slow vote that arrives after quorum still tightens the trust set rather than being silently dropped.
- **`FormedQc { qc, signer_indices }` is a transient internal type:** the on-wire `QuorumCertificate` has 96 opaque aggregate-sig bytes and *no* signer set. M7-d-6 will need to thread `signer_indices` into the header (likely as a bitmap, since the validator set ordering is canonical) — until then `verify_formed_qc` consumers must carry signer indices alongside. Adding the bitmap is a *breaking* wire change; that's why M7-d-6 is the last and breaking substep.
- **libp2p gossipsub `publish` + two-node QUIC:** right after `ConnectionEstablished`, `gossipsub::publish` often returns **`PublishError::NoPeersSubscribedToTopic`** because mesh graft / SUBSCRIBE exchange has not completed. For devnet-sized meshes, lower **`mesh_n` / `mesh_outbound_min`** than defaults and call **`add_explicit_peer(&peer_id)`** on `ConnectionEstablished`. Still queue at least one pending **`Vote`** payload and **retry `publish` after every `Swarm` poll** until it succeeds (or a non-retryable error), so votes are not dropped when the producer commits before the follower finishes gossipsub handshake.

Unit test `p2p_identity_tests::load_or_create_stable_peer_id`. **`docs/devnet.md`** subsection **P2P host identity**. **Not in this slice:** Kademlia DHT, Noise-on-TCP (QUIC uses TLS), tying libp2p host key to consensus validator identity (BLS remains the signing identity for blocks/votes).
